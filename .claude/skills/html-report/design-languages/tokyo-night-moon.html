Tokyo Night Moon, terminal-native

A mono-led report register for documents that look like they were written in the editor they describe. Fira Code for body, headings, and code — hierarchy through size and weight only, ligatures everywhere.

status: candidate audience: engineer-internal v1
LockedFira Code 600 h1 with letter-spacing -0.03em, magenta <em> wrapped in <> brackets, mono caps eyebrow prefixed with magenta $, blinking cyan square dot, bg-dark → bg-editor gradient with magenta hairline.
FreePadding, lede max-width, hero footer as .meta-row key/values or .meta pill chips.
01

Personality

Moon is the developer-in-a-dark-room-at-2am variant. The whole document looks like terminal output that someone formatted nicely. Every glyph is monospaced; hierarchy comes from size (30 / 20 / 16 / 14 px) and weight (300 / 400 / 500 / 600). There is no sans, no serif, no escape from the grid.

Moon uses the folke/tokyonight.nvim palette — deeper and slightly more cosmic than Storm. The base is #222436 (less blue, more purple-navy). Magenta (#c099ff) is the lead accent — it sits where blue sits in Storm. Cyan (#86e1fc) is secondary. Orange (#ff966c) is warn — warmer than Storm's #e0af68, more like the orange of a TTY cursor than VS Code.

Code blocks should bleed seamlessly into surrounding text — when the difference between "prose" and "code snippet" is just a background tint, the document feels native.

Mono-only sounds restrictive; it isn't. Fira Code has 7 weights, contextual ligatures (!=, =>, >=, ->), and stylistic alternates. The constraint is the brand: a Moon report is instantly recognizable because everything inside it is set in the same monospaced grid, like a screenshot of a well-curated .config/ directory.

02

When to pick this

Pick this when

CLI tooling docs, shell scripts, configuration walkthroughs, terminal-output postmortems. Reports where ~40%+ of the content is code or shell. Internal docs for developer-tooling teams who self-identify with their .dotfiles. Documents read inside a tmux pane.

Don't pick this when

The audience includes non-engineers (PMs, designers, executives). The report is more narrative than technical — prose-heavy postmortems lose rhythm in mono. The reader will skim — mono uniformity makes skimming harder than weighted prose. Sunlit-room delivery — go to tokyo-night-day.

03

Locked brand stack

Load exactly this Google Fonts URL — Fira Code is the only font in the document.

<link href="https://fonts.googleapis.com/css2?family=Fira+Code:wght@300;400;500;600;700&display=swap" rel="stylesheet">

Font stack variable

/* exact value — verbatim */
--mono: "Fira Code", "JetBrains Mono", "SF Mono", "IBM Plex Mono", Menlo, monospace;

There is no --serif, no --sans. Every element uses var(--mono).

Ligature settings

body {
  font-variant-ligatures: contextual;
  font-feature-settings: "calt", "ss01", "ss03";
}

Ligatures are on everywhere — !=, =>, ->, <=, ===, &&, || all collapse. ss01 activates the open a; ss03 gives the rounded l. Disable them anywhere and Moon stops feeling like itself.

Voice rules

  • Headings carry markdown-style prefixes: h3 renders ## in magenta, h4 renders ###. This is non-negotiable — it is the document's signature.
  • Section numbers prefix with // in fg-faint and the number itself in cyan.
  • The eyebrow leads with $ in magenta — the shell prompt.
  • The blinking square cyan dot in the eyebrow is the only animation in the language. A 1.4s blink, two steps.
  • Pull quotes are wrapped in """ top and bottom — triple-quoted strings, in cyan.
  • Italics are rare. Use weight contrast (400 → 500 → 600) instead.
  • Inline links default to cyan, hover to magenta — the inverse of normal hover (cyan is calmer than magenta here, so hover should escalate the loudness).

05

Anti-patterns

  • No proportional fonts anywhere. Adding Inter, Geist, or any sans for "readability" breaks the brand — Moon's whole point is the monospaced grid.
  • No ligature disabling. Removing contextual ligatures turns Fira Code into a generic mono and loses the Moon signature.
  • No italic emphasis in body prose. Hierarchy is weight-only (400 → 500 → 600).
  • No drop shadows beyond 0 8px 24px rgba(10, 12, 22, 0.4). The pseudo-CRT magenta hairline above each card is the language's lift, not shadow weight.
  • No multiple animations. The blinking cyan eyebrow dot is the only motion. Adding spinners or other blinkers is forbidden.
  • No emoji, no icons. The markdown-style heading prefixes (##, ###) and arrow markers (▶, ▸, ▾) carry the affordance.
  • No purple and blue as co-primary. Blue exists only in code tokens (function names). Moon's identity is magenta + cyan.
  • No pure-black backgrounds. The base is the exact #222436 indigo-purple.

07

Diagram philosophy

Diagrams in Moon are mono-text schematics — light Unicode ┌─┐ │ └─┘ frames in SVG <text> blocks, with labels styled like real code (trigger(), store.save()) and inline // comment annotations between messages. Fira Code with contextual ligatures so ─> and // read like a source file. Role shows through ink: default keeps neutral fg, hot shifts to magenta, external to cyan, warnings to orange — no fills, no fancy SVG primitives.

SVG typography

Each diagram is a single <text> with xml:space="preserve" and stacked <tspan x="10" dy="1.3em"> lines. Fira Code with font-variant-ligatures: contextual so ─> and ─ ─ render with the font's intended joins. Color and weight come from nested <tspan class="...">:

.ascii      { font: 500 14px "Fira Code", monospace; fill: var(--fg-primary);
              white-space: pre; font-variant-ligatures: contextual; }
.ascii .hot  { fill: var(--magenta); font-weight: 700; }
.ascii .ext  { fill: var(--cyan); }
.ascii .com  { fill: var(--fg-secondary); }
.ascii .step { fill: var(--cyan); letter-spacing: 0.10em; }
.ascii .warn { fill: var(--orange); font-weight: 600; letter-spacing: 0.08em; }
LockedLabels read as code: trigger(), store.save(), webhook("job.completed"). Inline comments prefixed // in fg-secondary. Step counters <NN> in cyan. Alert callouts in orange snake_case (e.g. retry_policy_engaged).
FreeWhether to use step counters on small diagrams, exact letter-spacing (0.08–0.14em), label arity in the function-call labels.

Alert region pattern

Failure paths wrap in a literal ┌─── retry_policy_engaged ───┐ … └────┘ frame in orange. The frame is text — same box-drawing glyphs as the regular boxes, just inked orange. The inline callout opens with a small block (■) instead of a circle (Moon avoids circular glyphs in diagrams), followed by the snake_case label and a // backoff … comment. See the sequence diagram above between steps <03> and <04>.

LockedFrame uses orange ink on light box-drawing glyphs. Inline callout uses a square block ■ (no circles), a snake_case orange label, and a //-prefixed mono detail.
FreeFrame width, label copy, presence of the inner // backoff … comment.

Arrow vocabulary

Single-line ─> for default and hot transitions (Fira Code ligatures render this as a single connected arrow). Returns shorten to <─ ─ (spaced en-dashes) in fg-faint. Async fan-outs use the same ─ ─ ─ spaced form. Warn arrows live inside the orange frame and inherit its ink.
