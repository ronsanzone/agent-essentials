Design Language · Docs Register

Stripe Docs, built for reference

A developer-docs register — dense, scannable, code-heavy. For API reference, integration guides, runbooks, and engineering reports that get re-read, not read once. Optimized for utility over airiness.

01

Personality

Stripe Docs is the API-reference register: white background, dense body text, code blocks everywhere, link-blue (#008cdd) as the lead accent. This is the register a developer scrolls through with a terminal open in the other window. Utility over airiness, scannability over breathing room.

Reference docs are not read; they are scanned. Every visual decision should serve the scan.

Link-blue is the lead accent for a reason: it reads as a hyperlink, and reference docs are dense networks of hyperlinks. Purple is reserved as a secondary accent for special emphasis — pull quotes, highlighted sections — to keep the link-blue economy clean. Red (#dc2626) is the proper warn color because API errors carry a different weight than press-page caution: deprecations, breaking changes, 4xx/5xx responses.

02

When to pick this

Pick this when

API reference. Integration guides. Runbooks. Engineering postmortems. Any report with high code-block density and reference-style scanning patterns. Tables of methods, parameters, error codes.

Don't pick this when

Marketing-grade announcements — use stripe-press. Long-form essays — use stripe-blog. Reports that are read once and forgotten don't need this density.

Voice rules

  • H1 carries an italic em phrase in link-blue: <h1>Title, <em>italic phrase</em></h1>.
  • Display sizes are moderate — h1 at 32–44px, h2 at 28px. The docs register reads at desk-distance, not screen-projection.
  • Body line-height is 1.6 (vs press's 1.65) and font-size is 15px (vs press's 17px) — density beats airiness.
  • Code blocks get a 3px --link-blue left border — they're navigational landmarks.
  • Method pills use semantic color: GET = link-blue, POST = success-green, DELETE = warn-red, PATCH = amber.
  • Never bold body prose for emphasis — use italic or inline code.
  • Tables stripe even rows in --surface-1 — reference scanning wants the visual anchor.

Why red for warn

API errors carry weight that the press register's amber doesn't. 4xx / 5xx responses, breaking changes, deprecations, and failure paths all need a color that reads as "something went wrong" — not "be careful." Red (#dc2626) is the conventional choice in developer tooling and the right choice here.

Warning: Use red sparingly. Reserve .callout.warn for breaking changes, deprecations, security advisories, and known-broken states. Overuse of red causes alert fatigue and erodes its signal value.
05

Anti-patterns

  • No serif body. (Serif body belongs to stripe-blog.)
  • No oversized display headings — the press-page h1 sizes (88px) are wrong here. Reference docs read at 32–44px max.
  • No warm surfaces — no parchment, no cream, no beige. Cool only.
  • No purple as the lead accent — purple is the secondary accent in docs. Link-blue leads.
  • No amber warn — red is the docs warn color.
  • No drop-shadows on cards. Docs cards are flat: border + background only.
  • No body line-heights above 1.6 — that's the press register.
  • No emoji in headings or callouts.
  • No marketing-page gradients, no pull-out illustration cards, no testimonial slabs.

07

Diagram philosophy

Diagrams read as API reference cards. Boxes are pure rectangles (0px or 2px corners, never rounded), all labels in IBM Plex Mono, and node-internal HTTP method pills are drawn inline as SVG <rect> + <text> chips next to a monospace endpoint path. Connectors are 1.5px #008cdd with right-angle bends — wiring-diagram geometry, not curves — and chevron heads drawn as path strokes rather than triangle markers. Status codes (201, 429, 503) ride the arrows in small mono. Alert regions render as rectangular HTTP error-response panels with a 4xx / 5xx code header.

SVG typography

Diagrams in the docs register are mono everywhere — labels, endpoints, status codes, notes. Inter does not appear inside diagrams; the visual goal is "diagram = API reference card", and the reference voice is monospace.

Inline HTTP method pill

The signature trick: hot and external nodes carry an inline HTTP-method chip drawn as a small filled <rect> + white .pill-txt label, placed next to a monospace endpoint path. POST uses --success (green), GET uses --link-blue or --purple (external), DELETE uses --warn. The chip stays inside the SVG so the diagram is self-contained.

Alert region pattern

Failure paths render as HTTP error-response panels: a pure rectangle (no rounded corners) with a red header bar showing 4xx/5xx + status code, and a JSON-shaped body below. A small BACKOFF chip — white fill, red border, mono caps — sits inline with the retry policy detail.

LockedPanel: rx="0", --warn-soft fill, 1.5px --warn stroke. Header: 20px-tall --warn bar with white mono 4xx/5xx label + status code. Body: JSON-shaped err-body in dark red.
FreePanel height, exact JSON content, presence of the inline BACKOFF chip.

Arrow geometry

Default: var(--link-blue) at 1.5px with right-angle bends — wiring diagrams, not curves. Chevron heads are drawn as <polyline> paths (points="x1,y1 x2,y2 x3,y3") rather than triangle markers, so the head reads as a literal >. Return paths use --rule-strong. Async paths use stroke-dasharray: 4 3. Warn paths use --warn.
